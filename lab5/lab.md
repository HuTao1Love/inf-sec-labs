## Дополнительные вопросы
1. Какие типы уязвимостей лучше всего выявляются SAST?
	SAST - для поиска ошибок в структуре кода:
	- SQL-инъекции и XSS: прослеживание пути данных от ввода до исполнения.
    - Хардкод кредов: поиск паролей, ключей API и токенов, зашитых в код.
    - Использование небезопасных функций: например, `eval()` в JS/Python.
    - Ошибки криптографии: использование слабых алгоритмов шифрования (MD5, SHA1).
    - Переполнение буфера: особенно актуально для компилируемых языков (C/C++).
2. Какие типы уязвимостей можно найти только при помощи DAST?
	- Ошибки конфигурации сервера: неправильные заголовки HTTP, включенные методы (TRACE/TRACK).
	- Проблемы с сессиями: небезопасные Cookie (отсутствие флагов HttpOnly/Secure).
	- Ошибки логики аутентификации: уязвимости, возникающие только при взаимодействии компонентов.
	- Проблемы инфраструктуры: уязвимости в сторонних библиотеках среды выполнения или веб-сервера.
3. Какой анализ проводить первым: статический или динамический? Почему?
	**SAST**:
	- Экономия: ошибки исправляются дешевле на этапе написания кода.
    - Скорость: SAST можно запускать сразу после написания функции, не дожидаясь сборки и развертывания приложения.
    - Покрытие: SAST проверяет все ветки кода, включая те, которые трудно вызвать в работающем интерфейсе.
4. В каких случаях SAST даёт ложные срабатывания?
	- Особенно актуально для динамически типизированных языков
	- Не понимает контекст: например, не видит, что данные уже очищены внешним фреймворком или WAF.
    - Сложная логика: теряет путь переменной в цепочке из множества вызовов функций.
    - Универсальные правила: слишком строгие паттерны, которые не учитывают специфику конкретной бизнес-логики.
5. Как избежать ошибок при настройке DAST-сканирования?
	- Настройка аутентификации
	- Ограничение области: указать URL-адреса, чтобы сканер не ушел на сторонние ресурсы (например, ссылки на соцсети).
    - Безопасность данных: проводить сканирование в staging-среде, так как DAST может заполнять БД мусором или случайно удалить записи через формы.
    - Исключение деструктивных действий: отключить проверку функций удаления или сброса пароля.
6. Какие ограничения есть у SAST-сканирования по сравнению с DAST?
	- Отсутствие контекста среды: SAST не знает, на какой ОС и с какими настройками сервера работает код.
    - Зависимость от языка: SAST требует глубокой поддержки синтаксиса конкретного языка программирования.
    - Слепота к интеграциям: SAST плохо видит проблемы, возникающие на стыке приложения и базы данных или внешних API.
7. Как защититься от уязвимостей, которые не детектируются ни SAST, ни DAST?
	- Анализ зависимостей (библиотек) + локальное зеркало (nuget/pypi/etc)
    - Ручной пентест
    - Code Review
    - WAF (Web Application Firewall)
8. Какие open-source инструменты используются для SAST и DAST, и в чём их отличия?
	1. SAST
		- Semgrep
		- SonarQube
		- GitHub CodeQL
		- Линтер, встроенный в IDE/компилятор (например, анализаторы Roslyn в dotnet publish) - не совсем SAST, но...
	2. DAST
		- OWASP ZAP (Zed Attack Proxy)
		- Nuclei
9. Как часто следует проводить статический и динамический анализ кода в CI/CD?
	- SAST - при каждом пуше или МРе
	- DAST - при каждом деплое на тестовую среду или по расписанию
10. В чём разница между интерпретируемыми и компилируемыми языками с точки зрения анализа уязвимостей?
	- Компилируемые (C++, C#, JAVA): SAST ищет ошибки управления памятью (утечки, переполнения). Для анализа часто требуется сначала успешно собрать проект.
    - Интерпретируемые (Python, JS, PHP): SAST работает проще (анализ скриптов), но выше риск ложных срабатываний. Ошибки типизации выявляются сложнее без запуска.
11. Как можно использовать результаты DAST-сканирования для улучшения защиты веб-приложения?
	- Виртуальный патчинг: настройка правил WAF на основе найденных уязвимостей, пока разработчики готовят исправление.
    - Приоритизация: если DAST подтвердил уязвимость, найденную SAST, её приоритет становится критическим (так как она точно эксплуатируема).
12. Почему важно учитывать контекст уязвимости при интерпретации отчёта SAST?
	Например, проблемы в тестовом скрипте - низкий риск, а в проде - крит инцидент (хардкоды, sql инъекции, etc)
13. Можно ли полностью автоматизировать анализ безопасности кода, или требуется ручная проверка?
	Инструменты - только для поиск известных паттернов
	Человек - для отсеивания ложных срабатывания, анализа бизнес-логики, цепочек уязвимостей
