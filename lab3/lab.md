## Дополнительные вопросы
1. Кто такой владелец БД и какими правами он обладает? В чем его отличие от суперпользователя?
	**Владелец БД (Owner)** — это роль, которой назначено владение конкретной базой данных.
	- **Права:** Владелец обладает полными правами на все объекты внутри своей БД (таблицы, схемы, индексы), может изменять параметры базы и удалять её.
    - **Отличие от cуперпользователя:** В то время как полномочия владельца ограничены рамками конкретной БД, суперпользователь обладает неограниченными правами во всей СУБД. Он игнорирует любые проверки прав доступа, может управлять системными файлами и изменять глобальные настройки сервера.
2. Какие механизмы безопасности можно использовать в PostgreSQL для предотвращения несанкционированного доступа?
	- **Аутентификация на уровне хоста:** файл конфигурации `pg_hba.conf` регламентирует, какие пользователи могут подключаться к серверу с определенных IP-адресов.
    - **Шифрование паролей**
    - **Row-Level Security (RLS):** механизм защиты на уровне строк, позволяющий ограничить видимость данных в таблице в зависимости от текущей роли пользователя.
3. Как можно организовать многоуровневый контроль доступа в базе данных?
	Многоуровневый контроль реализуется через иерархию разрешений (Privilege Hierarchy):
	1. **Уровень подключения:** Право `CONNECT` к базе данных.
	2. **Уровень схемы:** Право `USAGE` на использование конкретной схемы (например, `public`).
	3. **Уровень объектов:** Права `SELECT`, `INSERT`, `UPDATE`, `DELETE` на конкретные таблицы или представления.
	4. **Уровень столбцов:** Назначение прав доступа к отдельным атрибутам (колонкам) таблицы.
4. Какие преимущества у ролевой модели в PostgreSQL перед традиционной моделью пользователей?
	- **Масштабируемость:** Права назначаются не индивидуально, а группам ролей (например, `itmo_readers`). Новым пользователям достаточно выдать членство в этой роли.
    - **Удобство администрирования:** Изменение прав одной роли автоматически обновляет доступ для всех её членов (наследование прав через `INHERIT`).
    - **Безопасность:** Минимизируется риск ошибок при ручной настройке доступа для большого количества сотрудников.
5. Как можно защитить PostgreSQL от атак методом подбора пароля (brute-force)?
	- **`auth_delay`** - использование модуля, добавляющего задержку перед ответом сервера после неудачной попытки входа.
    - **Fail2Ban**
    - **SSL-сертификаты** - использование аутентификации по клиентским сертификатам вместо классических паролей.
6. Какие меры безопасности нужно учитывать при передаче конфиденциальных данных между базами?
	- **Шифрование трафика (SSL/TLS)** - гарантирует защиту данных от перехвата в процессе передачи по сети.
    - **Принцип наименьших привилегий** - технический пользователь, осуществляющий связь между БД, должен обладать минимально необходимым набором прав для выполнения конкретной транзакции.
